Design

D5: X発信戦略（投稿の定型化とテーマ体系）

作成日: 2026-06-15 位置づけ: 定款・柱3「進化する独自指標で多角分析し、Xユーザーの興味を引く切り口で投稿し続ける」の実装設計。分析切り口カタログで有望と検証した切り口を、反復可能な投稿テンプレートに落とす。正本の指標は D4/D4.5、データは D3、品質ゲートは D2/D6。 鉄則（定款から継承）: 確証を持てない数字は投稿に一切流さない。status=ok のclaimだけが投稿に出る。誤った数字を自信ありげに出すこと＝最大の事故（花王 36年を16年と言う類）。

0. 前提と制約（バッジ無しアカウント）

制約 内容 設計上の含意
140字制限 バッジ無し＝日本語約140字（CJKは加重2）。本文だけで複雑な分析は載らない 画像でデータを運ぶのが主戦術（§1）
API投稿不可 無料APIプランでは投稿不可 Playwright（ブラウザ操作・セッション再利用）。既存 x_poster.py を流用
画像は文字数外 画像は最大4枚、本文文字数を消費しない。URLは t.co 23字固定 表・チャートを画像化＝実質的な文字数制約の突破口
スレッドは手動連結 API無しでもPlaywrightで「投稿→自分にリプ」を繰り返せばスレッド化可能 長い話はスレッド型（§1-C）。各ツイートは140字＋画像

→ 設計の核: 「140字の本文＝フック（興味喚起）」「画像＝データ（証拠）」に役割分担する。本文に数字を詰め込まない。

1. 制約の突破：投稿フォーマット3類型

類型 構成 用途 文字数会計
A 単発フック型 本文140字のみ 問いかけ・気づき・1つの数字 本文≤140
B 画像主役型（主力） 本文140字（フック）＋ 画像1〜4（ランキング表/チャート） ランキング・比較・シミュレーション 本文≤140＋画像
C スレッド型 親（フック＋画像）→ リプ連結（各140字＋画像） 1銘柄の深掘り・シミュレーションの手順 各ツイート個別に≤140

主力は B（画像主役型）。140字では「○○な増配株トップ5」とフックだけ書き、5銘柄の数字・グレード・status は画像の表に載せる。画像はサーバ側で生成（HTML/matplotlib→PNG）し、Playwrightで添付する。

2. 投稿の定型化（テンプレート構造）

投稿物 ＝ テンプレート × データクエリ × 品質ゲート。1投稿を固定パーツに分解する。

パーツ 役割 ルール
フック（1行目） スクロールを止める。問い・意外性・数字1つ 投稿の生命線。テーマごとに型を用意
主張（claim） 中身。「○○は△△」 status=ok のみ。数字は出典・as_of が紐づくものだけ
根拠 画像（表/チャート）or 簡潔な一文 画像に銘柄・指標・グレード・status を明記
締め 問いかけ/次回予告（CTA） エンゲージ誘導。煽らない
固定フッタ 出典・免責・ハッシュタグ 免責は必須（§5）。#日本株 #増配株 #高配当 等

テンプレートは themes/*.yaml（テーマ定義）で管理し、差し込み変数（銘柄・数値・グレード）は computed_metrics / dividend_scores から取得。本文に出る数字は必ずDB由来＝手打ちしない（捏造防止）。

3. テーマ体系（切り口カタログ駆動）

テーマ ＝ 切り口カタログの具体化。1切り口から複数テーマが生える。

テーマ 元の切り口 形式 必要データ（status=ok前提） 想定頻度
目標逆算シミュレーション（つみたて→将来配当） ①YoC×持続性 B/C YoC実績・実証済み増配率レンジ 月次・看板
YoC実績（N年前に買っていたら今利回り○%） ①A B price_history(2000〜)・DPS 週次
増配の質（本物の増配株 vs 一過性の罠） ①B B/C DPS倍率・EPS倍率・配当性向 週次
連続増配ランキング 既存＋override B streaks（N+表示・result_override） 週次
配当性向の余力 / DOE ①C B 配当性向・DOE・FCFカバレッジ 隔週
割安×増配（バリュエーション） （D4財務） B PER/PBR・グレード（valuation） 隔週
高配当ランキング （D4配当） B 配当利回り・配当grade 週次

運用: 1テーマ＝1 themes/*.yaml。新しい切り口がカタログに増えたらテーマを追加する（柱2/柱3の蓄積が投稿バリエーションに直結）。

4. 看板テーマ詳細：目標逆算シミュレーション

ユーザー例：「毎月3万円のつみたてで将来月5万円の配当を得る方法」

万人に刺さる「目標→道筋」型。切り口①（YoC×増配持続性）を物語化する。

4.1 入力パラメータ

  • 月額つみたて額（例 30,000円）
  • 目標（月配当 50,000円 ＝ 年60万円、または「達成年数」）
  • 期間（例 20年）
  • 想定増配率・想定利回り（捏造しない＝§4.3）

4.2 計算モデル（2つの見せ方）

  • (a) 順算: 「月3万 × N年・増配率r%・利回りy%」→ 将来の年間受取配当
  • (b) 逆算: 「月5万円の配当に必要な元本・年数」→ つみたて額から逆算
  • 要素: 積立（元本累積）＋ 配当再投資（複利）＋ YoC上昇（増配で取得利回りが育つ＝切り口①の主張そのもの）
  • 税の扱いを明示（税引前/後、NISA前提の有無）

4.3 データ連動（捏造しない仕掛け）

シミュレーションの肝は想定増配率。ここを適当な数字にすると定款違反。

  • 増配率は、切り口①で実証した「本物の増配株」（EPS牽引・DSS高）の実績レンジから採る（例：過去5年のEPS牽引増配株の中央値）
  • 「夢の数字」を避け、保守/標準/強気の3シナリオで幅を見せる
  • 実在の増配株（status=ok・配当grade A）を具体例として併記（「例：○○は5年で取得利回りが3%→X%」）

4.4 出力（画像主役）

  • 本文（≤140字）: 「月3万円の積立、増配株ならいつ月5万円の配当になる？」＋ 結論1つ
  • 画像1: 積立＋配当成長カーブ（年×受取配当、3シナリオ）
  • 画像2: 根拠となる実在増配株のYoC実績表（銘柄・YoC・EPS倍率・grade）
  • スレッドにする場合: 親=結論、リプ1=前提、リプ2=実例、リプ3=注意点

4.5 必須の但し書き

将来予測は前提次第。固定フッタで「これはシミュレーションであり将来の成果を保証しない／投資判断は自己責任」を明示（§5）。特定個人向けの助言はしない（一般教育コンテンツとして設計）。

5. 品質ゲート連動（定款の鉄則の実装）

投稿は D2/D6 の品質判定に従属する。ゲートを通らない数字は本文にも画像にも出さない。

ゲート ルール
status フィルタ 投稿に使う全claimが status=ok。missing/insufficient は出さない
N+ 表示の徹底 左打ち切りの連続年数は「N年以上」。裸の数字で言い切らない（花王36→16事故の再発防止）
claim別グレード整合 配当grade=A だが capital=D の銘柄は「配当の話」だけ。財務優良と混同させない
出典・as_of 必須 result_override由来の数字は出典・基準年を画像/フッタに記載。定義差（「上場前から起算」等）も注記
自動停止（D6接続） 照合レポートで override乖離・status分布異常を検知したら当該テーマの投稿を自動停止
免責フッタ 「投資助言ではない／自己責任」を全投稿に固定。シミュレーション系は「将来を保証しない」を追加

不変条件: findexは「自信ありげな誤った数字」を絶対に出さない。確証が持てないなら投稿しない（沈黙は許容、誤発信は不可）。

6. 運用（Playwright・スケジュール・学習）

6.1 投稿実行

  • Playwright・セッション再利用（既存方針）: ~/.findex/x_session.json を再利用しレート制限を回避。初回/失効時のみログイン
  • 既存 x_poster.py（ログイン/投稿/スレッド/画像添付のセレクタ判明済）を流用。テーマ定義・画像生成・品質ゲートを上流に足す
  • 投稿前にドライラン（本文・画像・通過したゲートを標準出力で確認）→ --publish で実投稿

6.2 スケジュール

  • cron（朝9:00/昼12:00/夕18:30 を基本）。テーマを曜日・時間に割り当て（例：週次ランキングは特定曜日）
  • 1日の投稿数・同一テーマ間隔の上限を設定（連投・重複の抑制）

6.3 投稿ログと学習ループ

  • post_log（§7）に テーマ・銘柄・claim・画像・投稿URL・エンゲージメント を記録
  • 将来：エンゲージメントを集計し効くフック/テーマを強化（柱2/柱3の進化）。初期は手動レビューで十分

7. データモデル追記提案（D3への差分）

X発信に必要なテーブル（D3 data-model に追加）。

テーブル 役割 主なカラム
themes テーマ定義のレジストリ theme_id, name, angle_ref, format(A/B/C), enabled
post_queue 生成済み・投稿待ち id, theme_id, body, image_paths, claims(json), gates_passed, status(draft/approved/posted/blocked)
post_log 投稿実績・効果測定 id, theme_id, posted_at, tweet_url, body, claim_refs, engagement(json)

post_queue.claims は使用した数字とその status/source/as_of を保持（事後監査可能性＝何を根拠に投稿したか追跡できる）。

8. 段階導入（実装フェーズ Phase 6）

実装は実装計画 Phase 6（優先度低・他フェーズ安定後）。中でも段階を踏む。

  1. MVP: 1テーマ（連続増配ランキング・画像主役B）を手動承認付き（draft→人が確認→publish）で開始
  2. 看板: 目標逆算シミュレーション（§4）を追加。シナリオ幅・実例・免責を実装
  3. 自動化: 品質ゲートと自動停止が緑で安定してから cron 自動投稿へ
  4. 学習: post_log の効果測定→テーマ調整
