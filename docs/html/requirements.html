Findex 要件定義書 v2

作成日: 2026-06-14 ステータス: ドラフト（§12 のオープン事項が未決）

0. このドキュメントについて

旧実装（../back_findex）を切り離し、ゼロから作り直すための要件定義書。 旧実装は 計画ドキュメントは良質だったが、実装コードが破綻していた:

  • db.py（31KB）— マイグレーション11世代の継ぎ足しで新旧テーブル定義が混在
  • cli.py（39KB）— コマンドごとに生SQLが埋まったモノリス
  • フェッチャーが直接スコアまで書く二重パイプラインで値が食い違う
  • yfinance依存によるデータ品質の破綻（← 再構築の最大の動機）

旧 plans/ docs/ は設計資産として有効。本書はそれらを統合し、何を引き継ぎ・何を捨て・何を直すかを一本化したもの。

引き継ぐ方針（確定済み）: - ✅ スコアリング設計思想（18指標・197点→100点換算 を踏襲） - ✅ 収集済みデータ（~/.findex/db/findex.db）を移行して再利用 - ✅ X自動投稿機能

1. 背景と再構築の理由

最大の課題: データ品質・正確性

Xに投稿する数字が間違っていることが最大の事業リスク。旧実装では yfinance 依存により、財務データの欠損・配当履歴の歯抜け・ストリークの誤算出が頻発した。

代表例（花王 4452）:

公表値（正） 旧DB（誤）
連続増配年数 36期 26期

データ起点（yfinance/J-Quants は2000年前後が下限）でストリークが左打ち切りされ、誤った年数を出していた。これは「ほぼ嘘の情報」であり、信頼性を根本から損なう。

→ 正確性を最優先にデータ層を再設計する。速度・網羅性はその次。

2. プロダクトゴール

日本の全上場普通株式（約3,750銘柄、ETF/REIT除く）のデータベースを構築し、 増配継続株を多角的にスコアリングした結果を、定期的にX（Twitter）へ自動投稿する。

個人投資家が毎日手を動かさなくても、信頼できる増配株分析がXに流れ続ける状態が完成形。

3. スコープ

やること

  • 全上場普通株の銘柄マスター・株価・配当・財務データの保持と更新
  • 18指標スコアリングとランキング（CLI）
  • 連続増配/非減配年数を公表値と一致する精度で算出
  • X自動投稿（テーマローテーション、品質ゲート、二重投稿防止）
  • 日次・半年次の運用自動化

やらないこと（アンチ要件）

  • リアルタイム株価（日次終値で十分）
  • 売買執行・自動売買
  • Web公開（将来検討。当面は個人利用・SQLite）

4. 機能要件

F1. データ基盤

  • 全上場普通株（約3,750銘柄）の銘柄マスター・株価・配当・財務を保持
  • 配当は 会計年度別の年間1株配当（dividend_annual）を正準データとし、遡れる銘柄は1989年まで遡る
  • 上場日・設立年月日を銘柄マスターに保持（「2000年以前から上場している会社」を正確に特定するため。旧実装はこれを後回しにして判定が近似のままだった ← 再実装のきっかけ）

F2. スコアリング（18指標・100点満点）

  • 既存 rules.yaml の18指標体系を踏襲（§8）
  • 連続増配/非減配年数を公表値一致精度で計算
  • データ起点で打ち切られたストリークは「N年以上（N+）」と表示（嘘の数字を出さない）
  • 大型株・金融銘柄の動的指標入れ替えルール（§8参照）

F3. CLI

  • findex rank — ランキング表示（API不要・即時。フィルタ: market/sector/min-yield/min-no-cut/top/out）
  • findex update — 日次更新（株価のみ、約5分）/ --quarterly（財務）/ --dividends（配当履歴）
  • findex check <code...> — 個別銘柄の詳細確認
  • findex post <theme> — X投稿（--dry-run で品質ゲート＋生成のみ）
  • findex setup — APIキー設定

F4. X自動投稿

  • 1日1〜3スレッド、テーマローテーション（旧実装に21テーマの文面生成器あり、流用可能）
  • 投稿履歴テーブル（post_log）で二重投稿防止（本文SHA256、30日窓）
  • 品質ゲート: golden test（花王=36年 等）が通らない間は投稿を拒否
  • 投稿失敗時はリトライしない（翌日の枠で回復。凍結リスク最小化）
  • 投稿方式は Playwright（ブラウザ自動操作）で確定。X API は使わない（Freeティアでは実際に投稿できなかった実績あり）。.env の X_EMAIL/X_PASSWORD で自動ログインし、セッションを ~/.findex/x_session.json に保存して再利用（レート制限回避）。旧 x_poster.py を流用

F5. 運用自動化

  • 日次: DBバックアップ → 株価更新 → 指標再計算 → スコア → X投稿（launchd/cron）
  • 半年次: 配当イベント全銘柄再取得 → 正準系列（dividend_annual）再構築 → 整合性チェック
  • 全ジョブの実行ログを run_log に記録
  • 破壊的DB操作の前に必ずバックアップ（findex.db.bak-YYYYMMDD）

「2000年問題」の確定見解（複数回調査済み）

  • J-Quants は2000年前後が起点（無料プランは12週遅延で日次更新に使えない）
  • yfinance の配当は1999年9月以降のみ（それ以前は存在しない）
  • EDINET は2008年〜のXBRL（財務諸表の検証用。配当ストリークの長期遡及には使えない）
  • 2000年以前の無料一括データは存在しない。haitoukin が唯一の準一括ソース、それ以外は各社IR/国会図書館で個別対応
  • 有料ルート（日経NEEDS・Compustat・Datastream）は法人/大学契約前提で個人には非現実的

→ 長期ストリークは「収集済みバックフィル＋公表値オーバーライド＋N+表示」で正直に扱うのが現実解。根本対策（全銘柄の信頼ソース移行）は将来の J-Quants 契約次第（§12 オープン事項②）。

6. データ品質要件（最重要・新設）

旧実装が踏んだ「9つの地雷」をハード制約として明文化する。詳細は ../back_findex/plans/02-domain-knowledge.md。

# 地雷 新実装での制約
1 年間配当とイベント配当の混在 粒度の違うデータは別テーブル。年間値は dividend_annual、イベントは dividend_events。ストリーク計算は前者のみから
2 暦年集計は決算期変更で壊れる 必ず会計年度（4月始まり）で集計。fiscal_year = year if month>=4 else year-1
3 株式分割の調整基準がソース間で不一致 バックフィル接合時に境界異常チェック（前年比10倍超/1/10未満は調整漏れ疑い→そのバックフィルは捨てる）。NTT/SBG/電通総研は公表値オーバーライド
4 yfinance配当の年度歯抜け（210銘柄） 年度が連続しない箇所でストリーク打ち切り＋N+表示（安全側）
5 機械計算と公表値の定義差（±1〜2年） streak_overrides を持ち、公表値 > 機械計算のときだけ昇格
6 2000年以前データの入手性 収集済みCSVを使用。再スクレイピング不要
7 上場日の代用品は2000年floor 真の上場日は kabutan等から取得（yfinanceの最古日付は信用しない）
8 X投稿の規約・品質リスク 品質ゲート必須。投稿失敗はリトライしない
9 細かい知見 WALモード+busy_timeout、合併銘柄はフィクション扱い、破壊操作前バックアップ 等

品質保証の仕組み

  • golden test: 公表値（花王36年等、トップ20）と機械計算の突合テストを CI/投稿前に実行
  • 品質ゲート: golden test が落ちている間は X投稿を自動拒否
  • N+表示: streak_is_censored=1 の銘柄は「N年以上」で表示し、断定しない

7. データアーキテクチャ方針

設計原則（旧実装の破綻要因の裏返し）

  1. 一方向のデータフロー。取得層 → 導出層 → 評価層 → 出力層。後段は前段のテーブルだけを読む。逆流・ショートカット禁止
  2. 粒度の違うデータは別テーブル（地雷1）
  3. 指標計算は1モジュールに集約（ストリーク・CAGR・信頼性の計算は1箇所のみ）
  4. 全テーブルに出所（source）と更新時刻を持たせる
  5. DBアクセスと純粋計算を分離（テスト容易性）

データフロー

[取得層]
JPX Excel ──月次→ stocks（マスター＋上場日・設立日）
kabutan ──初回→ stocks.listing_date / founded_date
yfinance prices ─日次→ price_history
yfinance dividends ─半年→ dividend_events
yfinance financials ─四半期→ financial_snapshots（年度別・履歴保持）
haitoukin/IR/手入力 ──→ dividend_annual（source別）
公表ランキング・各社IR ──→ streak_overrides

[導出層]
dividend_events ─builder→ dividend_annual（source='events'）
dividend_annual + overrides + financial_snapshots + price_history
        ─compute→ computed_metrics（派生指標の唯一の出口・1銘柄1行）

[評価層]
computed_metrics ─scorer(rules.yaml)→ dividend_scores

[出力層]
dividend_scores + computed_metrics → CLI rank / 投稿文面生成
投稿文面 → 品質ゲート → X → post_log

テーブル構成（新スキーマ）

stocks / price_history / dividend_events / dividend_annual / streak_overrides / financial_snapshots / computed_metrics / dividend_scores / post_log / run_log / rule_versions / schema_version

詳細は以下に分割（正本は findex/db/schema.sql）: - docs/design/data-model.md — テーブル定義（各カラム・関連・地雷メモ） - docs/design/data-workflow.md — データワークフロー（更新サイクル・構築アルゴリズム・移行）

旧 stock_fundamentals / scores は二重パイプラインの遺物のため新DBでは作らない。

8. スコアリング指標（18指標・197点満点→100点換算）

100点は「理想の増配株」。平均的な優良株は40〜60点。定義は rules.yaml（ドメイン資産・そのまま流用）。

配当継続性（最重要）

指標 重み 満点条件
連続非減配年数 2.0 17年以上
減配信頼性スコア 1.5 過去20年で減配0回
連続増配年数 1.5 17年以上
配当成長10年CAGR 1.2 年8%以上
配当成長5年CAGR 1.0 年30%以上

配当安全性

指標 重み 満点条件
FCF配当カバレッジ 1.5 FCFが配当総額の2倍以上
予想配当性向 1.5 35%以下
EPS成長5年CAGR 1.0 年15%以上
売上高5年CAGR 1.0 年7%以上

財務健全性

指標 重み 満点条件
ROIC-WACC 1.3 10%以上
ROE 1.2 20%以上
自己資本比率 1.2 80%以上
営業利益率 1.0 20%以上
有利子負債比率 0.8 10%以下

バリュエーション

指標 重み 満点条件
利益剰余金配当倍率 1.3 10倍以上
配当利回り 1.2 3〜7%（7%超はペナルティ）
ミックス係数 0.8 PER×PBR ≤ 22.5
ネットキャッシュPER 0.8 10倍以下

動的指標入れ替え

  • 大型株（時価総額1兆円以上）: ROIC-WACC→利益剰余金配当倍率、ネットキャッシュPER→ミックス係数
  • 金融銘柄（銀行・リース・保険）: 自己資本比率→格付け、ROIC-WACC→利益剰余金配当倍率、ネットキャッシュPER→ミックス係数

9. 非機能要件

  • 正確性最優先: 投稿数字の誤りが最大リスク。速度・網羅性より正確性
  • DB: SQLite（~/.findex/db/findex.db）。WALモード + busy_timeout 30秒で更新ジョブとAPIの同時アクセスに耐える
  • 無料データソース優先。有料化（J-Quants Light等）の判断はユーザーに委ねる
  • yfinanceは壊れる前提: リトライ・バッチ間スリープ・フォールバックを設計に織り込む
  • 更新コスト: 日次（株価のみ）約5分、四半期（財務）約20分、半年（配当）約40分、初回フルスキャン2〜4時間

レート制限への対処（最大の運用ハードル）

全約4,000銘柄を一気に取得すると多くのサイトでレートリミットに当たる。fetch層は以下を必須要件とする: - 銘柄サブセット指定（--codes 7203,9433）で任意の部分集合だけ取得できること - バッチ分割＋バッチ間スリープ（yfinanceは2並列上限、礼儀スリープ1.5秒等。ソースごとに上限を設定） - チェックポイント／レジューム（途中失敗しても再開でき、取得済みは再取得しない） - レート制限検知時のバックオフ（429/401で指数バックオフ、上限超で中断して記録）

検証は「30社コホート」で回す（開発の前提）

実装の正しさは、毎回全銘柄ではなく例外になりうる約30社のコホート（data/verification_cohort.csv）だけで検証する。 - レート制限を避けつつ高速に反復できる - 失敗モード（バックフィル・歯抜け・分割不一致・1989年境界・合併・決算期・真IPO対照群）を網羅 - golden test の期待値は data/golden_streaks_zai_20260601.csv と上記コホート - 全銘柄スキャンは設計が固まり golden が通ってから1回だけ実行する

10. データ移行（収集済みデータの再利用）

本体: ~/.findex/db/findex.db（約218MB） - price_history 約181万行（2024-06〜）— 再取得可能だが時間がかかるので移行推奨 - dividend_history（→新 dividend_events）約87,791行 - dividend_annual 約54,188行（FY1989〜、backfill+overrides適用済み）— 必ず移行（source != 'events' の行と streak_overrides は再現困難） - 同内容は plans/data/*.csv にもエクスポート済み（保険） - 旧 scores 27,498行（〜2026-06）— バックテスト用途なら移行価値あり

移行時は新スキーマにマッピングし、dividend_annual の source 区分（events/haitoukin/ir/manual）と streak_overrides を確実に引き継ぐ。
