D2.7: 結果補正レイヤ（公表値オーバーライド）の一般化

作成日: 2026-06-14 ステータス: 設計（先行実装 streak_overrides あり） 親: 02_6-data-limits-and-impact.md。本書は「クラスB/D（マスターが取れない）」への現実解として、結果（derived result）だけを公表情報から補填する仕組みを設計する。 問い（ユーザー）: 2000年以前のマスターデータ収集は難しいが、評価に必要な"結果"（例: 連続増配30年）だけなら公表集計から集められる。それを補正データとして入れられないか？

0. 結論

できる。ただし対象を選ぶ。 findexには2種類のバックフィルがある:

種別 何を入れるか 例 対象
マスター補填（raw） 生の系列値 haitoukin で FY1989〜の年間配当 機械計算の入力を増やす
結果補正（override・本書） 導出済みの"結果"値 ZAi集計「連続増配28年」 機械計算を上書き/確定

本書は後者の一般化。既に streak_overrides（12件）・golden_streaks_zai（20件）として先行実装済み。これを「確報補正レイヤ」として制度化する。

1. 何が"結果補正"できて、何ができないか

公表されているのは要約された結果であって時系列ではない。だから補填可否はデータの性質で決まる。

評価軸 結果補正できるか 公表ソース 備考
連続増配年数 ✅ できる ダイヤモンドZAi / みんかぶ / 各社IR "結果"そのものが公表される代表例
連続非減配年数 ✅ できる 各社IR / ZAi 〃
減配信頼性（過去N年の減配回数） 🔶 一部 IR沿革 / 適時開示 減配の有無が文章で確認できれば
配当10年/5年CAGR 🔶 限定 ― クリーンな公表が少ない。raw補填（年間配当）経由が本筋
配当性向・自己資本比率・ROE等 ❌ 不要 ― 現在値で完結（D2.6 ④）。補正の必要なし
pre-2000の株価 / PER / PBR ❌ できない ― "結果"の公表が存在しない。点列データは要約で代替不能
長期トータルリターン 🔶 別物 ― 投稿フックとしては別途検討（D5）

核心: 補正できるのは「人間が要約して公表する性質の指標＝主に配当ストリーク」。 株価・バリュエーションは点列なので、要約結果での補填は原理的に不可。B/Dのうち救えるのは配当ストリーク系に集中する（＝findex核心の重み6.0そのもの。費用対効果が高い）。

2. 確報補正レイヤの設計

2.1 データ構造（現 streak_overrides を一般化）

フィールド非依存の汎用オーバーライドにする:

列 意味
code 銘柄
field 補正対象（consecutive_dividend_growth_years 等）。汎用化の鍵
value 公表された結果値
as_of_fiscal_year その値が「何年度時点」か（経年で増えるため必須）
source / source_url 出典（信頼ソースに限定）
definition_note 定義の but（後述の地雷対策。例「上場前から起算」）
confidence verified（2ソース一致）/ single（1ソース）
verified_at / verified_by 検証時刻・方法

2.2 採用ポリシー（機械計算との合成）

  1. 信頼ソース限定: ダイヤモンドZAi・各社IR・みんかぶ等の集計主体が明確なもののみ。個人ブログ等は不可（定款「確証を持てない数字は流さない」）。
  2. 昇格のみ: override ≥ 機械計算 のときだけ採用（公表値が機械計算より長い＝打ち切り救済）。機械計算の方が長ければ機械を信じる（公表の取りこぼし対策）。
  3. 打ち切りフラグ解除: override採用時は当該指標の is_censored=False（N+でなく確定値として出せる）。ただしその指標だけ。他指標は影響しない（claim単位・D2 §6）。
  4. 経年補正（ageing）: as_of_fiscal_year 以降は、機械計算で確認できた近年の連続分だけ自動加算（例: as_of2025=28年 + 2026も増配確認 → 29年）。再公表が出れば差し替え。古い override を放置して陳腐化させない。
  5. 二重チェック推奨: 重要銘柄（投稿頻度高）は2ソース一致で confidence=verified。

2.3 最大の地雷: 定義差（definition mismatch）

公表値を取り込む＝他者の定義を取り込む。これが最危険。 - 例（実在）: 小林製薬 4967 は「上場前から起算」する定義差で公表26年（機械計算と食い違う）。 - 「連続増配」は集計主体で定義が違う（据え置きの扱い・記念配当の除外・連結/単体・起算点）。 - 対策: ①definition_note に必ず根拠を残す ②投稿時は「ダイヤモンドZAi集計で連続増配28年」と出典を明示して断定を避ける ③機械計算値と併記してギャップを内部記録（D6の照合対象）。

2.4 カバレッジの現実

  • 公表ストリークランキングが載るのは有名な増配株の頭〜数百銘柄。下限band 1,365銘柄の長い裾は補正されない。
  • だが問題ない: 投稿で扱うのはまさにその有名銘柄群。結果補正は「投稿母数の質」を直接上げる（柱3に最も効く）。裾はN+で正直に扱えばよい。

3. 投稿（柱3）との相性 — むしろ強み

結果補正は品質リスクであると同時に投稿の武器になる: - 「ダイヤモンドZAi集計で連続増配28年（出典明示）」は、真実かつ信頼性が増し、かつ興味を引く。 - 出典付き＝定款の「確証を持てない数字は流さない」を満たしつつ、断定リスクを出典に移譲できる。 - → 結果補正レイヤは「品質ゲートを通すための保険」であり「フックの素材」でもある。

4. まとめと引き渡し

論点 結論
結果だけの補正は可能か 可能。配当ストリーク系に限り高い効果。先行実装(streak_overrides)を一般化
株価/PER/PBRのpre-2000 結果補正でも不能（点列は要約で代替できない）。N+/評価不能で正直に
最大リスク 定義差。出典明示・definition_note・機械値併記で管理
効果の所在 投稿母数の頭（有名増配株）を確定値化＝柱3に直接効く

→ D3（データモデル）: streak_overrides を汎用 result_overrides（field/value/as_of/source/definition_note/confidence）に拡張。raw補填(dividend_annual.source)と二層で保持。 → D4（指標）: 「機械計算 → 結果override（昇格のみ）→ N+ フォールバック」の合成順序と、定義差の併記ルールをデータ契約に明記。 → D6（検証）: override値と機械計算値のギャップを照合レポート化（定義差の早期発見）。 → 実装フェーズ: 信頼ソース（ZAi等）からの結果取り込みパイプライン（出典・as_of必須）。
