D2.5: 取得可能性スタディ — 実測findings

作成日: 2026-06-14 ステータス: 第1段（ライブプローブ）・第2段（旧DB集計）ともに実施済み 親: 02-data-integrity-framework.md（D2）。本書はD2辞書の各フィールドが「実際に集まるか」を実データで検証する。 手法: 使い捨てプローブ（.scratch/・gitignore）で J-Quants v2 / EDINET v2 に実アクセス。キーは back_findex/.env を参照。コホート1〜数社で surface を確認（レート制限配慮）。

0. 結論サマリ（第1段）

問い 実測結果
現在スナップショットの財務は1ソースで足りるか J-Quants v2 /fins/summary がほぼ充足（売上/営業益/純益/EPS/総資産/自己資本/現金/営業CF・投資CF・財務CF/発行株数/配当/予想配当/配当性向）。1コールで潤沢
長期時系列（配当ストリーク等）はライブで取れるか 取れない。この J-Quants 契約は約2年窓（2024-03-22〜2026-03-22）。10年/20年の遡及は不可 → 旧DB(FY1989〜)＋バックフィルが必須
入手難フィールドは本当に入手不能か 入手可能。EDINET XBRL から 利益剰余金・投資有価証券・社債及び借入金・支払利息 を実取得。低カバレッジの真因は「データ欠如」でなく「会計基準をまたぐパース未対応」
会計メタ（決算日・連結）は取れるか EDINETコードリストに同梱（XBRLパース不要）。決算日・連結有無・上場区分・業種が無料CSVで全社分
2000年以前の株価・PER・PBRは取れるか 取れない（§4.5）。2000年以前で保持できるのは配当のみ。株価はyfinanceでも2000-01-04が下限、歴史的PER/PBRは算出不能

→ 「EDINET一本足」ではなく、ソースは役割分担: 現在値=J-Quants、深い時系列=旧DB＋バックフィル、BS詳細・入手難=EDINET、会計メタ=EDINETコードリスト、識別=JPX。 → ただし価格時系列には明確な下限: 株価2000年・財務2008年。2000年以前は配当だけ（§4.5）。当初サマリは配当・現在財務に寄っており、価格の穴を後から明示した。

1. J-Quants v2（現行キーの実測）

ベース https://api.jquants.com/v2、認証 x-api-key。

エンドポイント HTTP 判定
/fins/summary 200 ✅ 使える。フィールド100超
/equities/bars/daily 200（範囲内） ✅ 使える。ただし契約範囲 2024-03-22〜2026-03-22 のみ。AdjC（調整後終値）AdjFactor あり
/listed/info 403 ❌ この契約では不可（識別は JPX Excel で代替）
/fins/dividend 403 ❌ この契約では不可（配当は /fins/summary の Div* で代替可）

/fins/summary が返す主フィールド（D2辞書への対応）

  • PL: Sales→revenue / OP→operating_income / OdP(経常) / NP→net_income / EPS DEPS(希薄化)
  • BS(一部): TA→total_assets / Eq→equity / CashEq→cash_and_equivalents / BPS
  • CF: CFO→operating_cf / CFI(投資CF) / CFF(財務CF) ※capex単独はEDINET
  • 株式: ShOutFY→shares_outstanding / AvgSh / TrShFY(自己株)
  • 配当: DivAnn(年間)→dps_annual / DivFY / Div1Q..3Q / FDivAnn→forecast_dps / PayoutRatioAnn→予想配当性向
  • 会計メタ: CurFYSt/CurFYEn(当期FY)→fiscal_period / CurPerType(FY/1Q..) / NxtFYSt/NxtFYEn
  • 連単: 連結が主、NC*(NCSales/NCTA/NCEq…)で単体も併存

履歴の深さ（重要）

KDDI で8件のみ＝FY2024〜FY2026の通期＋四半期。ライブの財務・配当は約2年しか遡れない。 → 配当ストリーク(17年)・配当10年CAGR・売上5年CAGR 等は J-Quants ライブ単独では不可能。旧DB(218MB, dividend_annual FY1989〜)とバックフィルが時系列の土台。

2. EDINET v2（入手難フィールドの実証）

ベース https://api.edinet-fsa.go.jp/api/v2、認証 Subscription-Key。

コードリスト（無料zip・XBRL不要）

searchdocument/codelist/Edinetcode.zip → 証券コード→EDINETコード 3,842件。 同梱列に 決算日・連結の有無・上場区分・資本金・業種 → 会計メタはここで全社取得可。

有報XBRL（KDDI 9433 / docID S100TN7R / 2024-06-20提出 / 2499行）

CSV列: 要素ID, 項目名, コンテキストID, 相対年度, 連結・個別, 期間・時点, ユニットID, 単位, 値

D2辞書フィールド 実測 例（KDDI・IFRS）
retained_earnings（利益剰余金） ✅ 5,522,578M
investment_securities（投資有価証券） ✅ 305,080M（個別BS）
interest_bearing_debt（社債及び借入金） ✅ 1,577,370M（非流動）＋短期分
interest_expense（支払利息）→cost_of_debt ✅ 8,813M
cash（現金及び預金） ✅ 個別 82,333M
current_assets（流動資産合計） ⚠️ 空振り。KDDIはIFRSで「流動資産合計」ラベルが異なる → キーワード辞書を会計基準別に持つ必要

確定した実装上の含意

  • 会計基準（JGAAP/IFRS/US）でXBRLラベルが変わる。利益剰余金（IFRS） のように接尾辞・科目構成が異なる。基準別のラベル辞書＋コンテキスト(CurrentYearInstant/連結個別)選別が必須。これを怠ると拾えず低カバレッジになる（＝旧実装の症状）。
  • 有報の探索は 提出日を日次スキャンして docTypeCode=120・csvFlag=1 を拾う（月末だけ見ると当たらない。旧 edinet.py のバグ）。3月決算は6月下旬窓。
  • beta のみ依然として外部に存在しない → 株価とTOPIX等指数から自前回帰で算出（フェッチ対象でなく導出対象）。

3. ソース階層の確定（D2 §3 の仮説 → 実測で確定）

役割 確定ソース 根拠
識別マスター JPX Excel（J-Quants /listed/info は現契約403） 実測403
会計メタ（決算日・連結） EDINETコードリスト（zip） 全社同梱を確認
現在スナップショット財務・配当・予想 J-Quants /fins/summary 1コールで潤沢を確認
直近株価（調整後） J-Quants /equities/bars/daily（2024-03〜のみ） AdjC確認・範囲制約確認
深いBS・入手難（投資有価証券/有利子負債/支払利息/利益剰余金/流動資産） EDINET 有報XBRL（基準別パース） 実取得を確認
長期時系列（配当ストリーク・各CAGR） 旧DB(FY1989〜)＋haitoukinバックフィル＋override J-Quantsライブ2年窓では不可
beta 株価×指数の自前回帰（導出） 外部に存在せず

「EDINET一本足リスク」は回避: 財務の現在値はJ-Quants、深い項目のみEDINET、と二系統に分散。J-Quantsの2年窓の弱点は旧DBが補う。相互補完で冗長性がある。

4. 第2段：旧DB(223MB)の現状gap実測（読み取り専用集計）

旧 ~/.findex/db/findex.db を read-only 集計（新規フェッチなし）。これが「いま何が欠けているか」の実数。

4.1 識別・属性（stocks 3,747件）

フィールド カバレッジ メモ
market / sector 100% JPX由来・問題なし
first_data_date 100% ただし自己参照で打ち切り検知に使えない（独立シグナルでない）
edinet_code 0% カラムは在るが未投入。EDINETコードリストで即3,842件埋まる（第1段で確認）
listing_date 0% 既知の宿題。kabutan等から取得が必要。打ち切り判定の鍵

4.2 財務（raw_financials 3,746件・yfinance由来）の非NULL率

フィールド カバレッジ 判定
total_assets / equity / liabilities / retained_earnings 99.5% ✅ 十分にある
current_assets 96.9% ✅
beta 95.3% ✅ 入手難ではなかった（D2 §2.7の想定を訂正）
long/short_term_debt 84.6 / 85.9% 🟨
operating_cashflow 75.8% 🟨
free_cashflow 70.6% 🟨
capital_expenditures 0% 🟥 一度も取得されていない＝FCFの真のボトルネック
investment_securities （列自体なし） 🟥 yfinanceに存在しない。EDINETで取得（第1段で実証済み 305,080M/KDDI）

D2への重要な訂正: 低カバレッジ指標（ROIC/FCF）の主因は beta ではなく ① capex 0% ② investment_securities 不在 ③ 複合指標が多フィールドの積で落ちること。 beta は95%あり問題でない。→ D2 §2.7 の「beta=入手難・低カバレッジ要因」は誤り。D4 のデータ契約に反映する。

4.3 配当時系列の実深さ（dividend_annual 3,210銘柄 / FY1989〜2026 / 54,188行）

source内訳: events(yfinance) 53,566行 / haitoukin(バックフィル) 622行。 銘柄ごとの最古FY分布（どこまで遡れているか）:

最古FY帯 銘柄数 含意
〜1990 49 深い履歴あり
1991-1999 9 〃
2000-2002（下限band） 1,365 データ起点で頭打ち＝打ち切り容疑。真の連続年数はもっと長い可能性
2003-2009 636
2010以降 1,151 多くは新しい上場＝真の開始の可能性（listing_dateで判別したい）
  • 打ち切り（streak_is_censored=1）= 1,039 / 3,746（27.7%）。約4分の1がN+表示。
  • バックフィルは 622行（≈58銘柄）止まり。下限band 1,365銘柄の大半は未補正＝正直にN+。
  • 配当データを持つのは 3,210銘柄。約537銘柄は dividend_annual が皆無（無配/取得漏れ）。

4.4 MVP母数（いますぐ投稿できる配当系claim）

  • 打ち切りでない確定ストリーク: 3,746 − 1,039 ≈ 2,707銘柄が「N+でない確定値」で語れる。
  • → 配当系の投稿は設計確定後すぐ数千銘柄分の母数がある。柱3は柱1の完成を待つ必要がない（D2 §6.5 MVPの裏付け）。

4.5 ⚠️ 時系列の下限（データ種別ごと）— 「2000年以前は配当だけ」

第1〜4節は「フィールドが取れるか」と「現在値」に寄っていた。価格時系列の穴を明示する（花王4452で実測）。

データ種別 実際の下限 2000年以前 根拠（実測）
年間配当(dps_annual) FY1989〜 ✅ ある haitoukinバックフィル。花王はFY1989=7.5円から連続
日次株価 2000-01-04〜 ❌ ない yfinance 4452.T max=2000-01-04（2000年以前0行）。旧DBは2024-06〜しか未収集だが再取得で2000年まで回復可
財務(EDINET XBRL) 2008〜 ❌ ない XBRL制度開始が2008
歴史的PER/PBR 実質2008〜 ❌ 算出不能 株価(2000〜)×歴史的EPS/BPS(EDINET 2008〜)の重なり。両方が要る

花王の「2000年以前」具体回答

  • 株価: 分からない（yfinanceも2000-01-04が最古、日本株の無料の2000年以前日次株価は事実上入手不可）
  • PER / PBR: 分からない（株価も過去のEPS/BPSも無いため算出不能）
  • 配当: 分かる（FY1989〜・バックフィル済み）。2000年以前で保持できるのは配当のみ

設計上の含意（定款「足りない部分を常に把握」に直結）

  • 2000年以前は配当ストリーク系claimのみ成立。価格・バリュエーション系claimはpre-2000は構造的に不能＝期間充足モデル(D2 §7)で「評価不能」と明示し、絶対に捏造しない。
  • 既存18指標のPER/PBR/利回りは現在値（スナップショット）しか使わないため、スコアリングには歴史的PER/PBR不要。影響を受けるのは「長期チャート」等の投稿フック（D5）で、それも2000年（株価）が下限と正直に扱う。
  • 旧DBの株価が2024-06〜しかないのは収集不足であって入手不能ではない → 実装フェーズで2000年まで再取得（yfinance、レート制限配慮）。

5. D2.5の結論と次への引き渡し

  1. ソースは役割分担で確定（§3）。EDINET一本足リスクは回避。
  2. 入手難フィールドの再定義（§4.2の訂正）: 真のgapは capex(0%)・investment_securities(不在)・edinet_code(0%)・listing_date(0%)。beta は問題でない。
  3. 時系列はライブ不可・旧DBが土台。下限band 1,365銘柄・打ち切り27.7%が「期間充足モデル」(D2 §7)の実母数。

  4. MVPは潤沢（確定ストリーク約2,700銘柄）。

  5. 時系列の下限を明文化（§4.5）: 株価2000年・財務2008年。2000年以前は配当系claimのみ成立、価格・バリュエーション系はpre-2000不能（捏造しない）。

→ D3（データモデル改訂）: edinet_code/listing_date/capex/investment_securities を埋める導線とカバレッジ追跡を設計に組み込む。株価は2000年まで再取得する前提でモデル化。 → D4（指標データ契約）: beta想定を訂正。capex取得後にFCF/ROICを再評価。Nullポリシー再設計。歴史的PER/PBRはスコアリング不要（現在値のみ使用）と確認。 → D5（X発信）: 価格・チャート系フックは2000年が下限と正直に扱う。 → 残宿題（実装フェーズ）: listing_date取得（kabutan）、capex取得（EDINET CF）、edinet_code投入（コードリスト）、investment_securities取得（EDINET）、株価2000年まで再取得（yfinance）。
