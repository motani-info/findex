# 設計: 2000年以前データ欠落への対処

**作成日**: 2026-06-14
**ステータス**: ⚠️ **位置づけ是正済み** — 本書はデータ完全性という大問題の**1事例（期間不足 × 配当）**にすぎない。全体像は [00-charter-and-data-integrity.md](00-charter-and-data-integrity.md) を正本とする。本書の手法（打ち切り検知・N+表示・公表値override・golden検証）は、より一般的な「期間充足モデル」「品質ゲート」の具体例として有効。
**関連**: [00-charter-and-data-integrity.md](00-charter-and-data-integrity.md)（ノーススター）/ [requirements.md](../requirements.md) / `../../back_findex/plans/02-domain-knowledge.md`（地雷集）

## 1. 問題定義

findex の中核スコアは**長期の配当履歴**に依存する（連続非減配17年・連続増配17年・減配信頼性20年・配当10年CAGR）。
しかしデータソースの下限は2000年前後:

- yfinance 配当 = 1999年9月以降のみ
- J-Quants = 2000年起点（無料は12週遅延）
- EDINET = 2008年〜（財務XBRL。配当ストリークの遡及には使えない）

2000年以前から上場していた会社は、配当系列が**下限年で左打ち切り**され、ストリークが過小評価される。

**実害の例（花王 4452）**: 連続増配 実際36年 → 旧DB 26年。「ほぼ嘘の数字」をXに出していた。

### 根本原因（実DBで確定）

| 事実 | 数値 |
|---|---|
| `stocks.listing_date`（上場日）の充足率 | **0 / 3747（0%）** — カラムは存在するが一度も埋められていない |
| `stocks.first_data_date`（DB内最古日）の充足率 | 3747 / 3747（100%） |
| 配当が FY2000 始まり | 308銘柄 |
| 配当が FY2001 始まり | 889銘柄（seam年度を捨てる地雷1のため2001が最大の壁） |
| haitoukin バックフィル済み | 58銘柄（FY1989〜） |
| 公表値オーバーライド | 12銘柄 |

`first_data_date` は「**自分が持っているデータの最古日**」を指すだけで、「**会社がそれ以前に存在したか**」は分からない（自己参照）。
打ち切りを検知するには「会社がいつから存在したか」という**独立した年齢シグナル = `listing_date`** が必要だったが、0%だった。
→ だから打ち切りに気づけず、誤った年数を黙って出していた。

## 2. 打ち切り判定式

```
is_censored(code) =
    (listing_year(code) < earliest_div_year(code))    # 下限年より前に上場 ＝ それ以前のデータが欠落
  AND
    (streak_start_year(code) == earliest_div_year(code))  # ストリークが下限まで遡って切れている
```

- `earliest_div_year` = `dividend_annual` の最小 `fiscal_year`（計算可能）
- `streak_start_year` = ストリーク計算の結果（計算可能）
- `listing_year` = `listing_date` 由来（**独立ソースが必要 = 欠けていた唯一のピース**）

補足:
- ストリークが下限band に達していなければ（途中で減配して2010年などから始まる）打ち切りではない → `listing_date` 不要
- `listing_year >= earliest_div_year` なら、データは上場時まで遡れている → **真の開始・打ち切りでない**（確定値を出してよい）

---

## 3. 3層の解決設計

### 第0層 — 絶対に黙って間違えない（最優先・データ取得不要）
ストリークが下限band（FY1999〜2002）に到達した銘柄は、**デフォルトで `streak_is_censored=1`** とし、表示・投稿では**常に「N年以上（N+）」**にする。
- これだけで「嘘の数字」は構造的に出なくなる（最悪でも正直なN+）
- 品質ゲートで担保（§7）

### 第1層 — 精度を上げる＝過剰なN+を消す（`listing_date` 取得）
N+ばかりではランキングが死ぬ。本当に2000年前後にIPOした会社は確定値を出したい。
- `listing_date` を取得し、判定式で「真のIPO」と「打ち切り」を切り分ける
- `listing_year ≈ earliest_div_year` → N+を外して**確定値**
- `listing_year < earliest_div_year` → 確定で打ち切り → 第2層へ

**取得方針**:
- **最終形は全銘柄（約3,750）取得が前提**（上場来パフォーマンス等、将来の分析にも使う）
- ただし**段階取得可**。打ち切り判定に効くのは配当が下限band（〜FY2002）で始まる**約1,200銘柄**なので、初回はここを優先してもよい
- ソース = **kabutan 銘柄プロフィール**（上場日・設立日を掲載）。J-Quants `/listed/info` にはIPO日が無い、JPX Excel にも無い
- 礼儀スリープ1.5秒。全銘柄で約100分、容疑者bandのみなら約30〜40分。**不変データなので初回1回限り**
- 代替: minkabu、日経会社情報、EDINET有報「沿革」（テキスト解析が必要で割高）

### 第2層 — 可能な範囲で実値に補正（既存資産で大半が完了済み）
1. **haitoukin バックフィル**（移行済み58銘柄、FY1989〜）。花王36年もこれで復元される。`dividend_annual` に `source='haitoukin'` で保持
2. **公表値オーバーライド**（12銘柄）。`streak_overrides` に保持し、**`公表値 > 機械計算` のときだけ昇格**（地雷5。古い公表値で機械計算を下げない）
3. **接合時の境界異常チェック**（地雷3）。前年比10倍超/1未満は分割調整漏れ疑い → そのバックフィルは捨てる。NTT/SBG/電通総研はこれで弾かれるため公表値オーバーライドで対処
4. 残り（1989年境界13銘柄など）は手入力オーバーライドで個別対応

## 4. データモデルへの影響

`stocks`（新スキーマ。`founded_date` を追加）:
```sql
listing_date    TEXT,   -- 上場年月日（kabutan等。打ち切り判定の独立シグナル）
founded_date    TEXT,   -- 設立年月日（補助）
first_data_date TEXT,   -- DB内最古データ日（導出値。単独では打ち切り判定に使えない）
```
`dividend_annual.source` ∈ {events, haitoukin, ir, manual}（粒度はFY。イベントと混在禁止 = 地雷1）
`streak_overrides`（code, growth_years, nocut_years, as_of_fiscal_year, source_url, verified_at）
`computed_metrics.streak_is_censored INTEGER`（1=「N+」表示。必須機能）

---

## 5. ストリーク計算の正準仕様（打ち切り込み）

入力: `dividend_annual` の (fiscal_year, dps) 昇順 + `streak_overrides` + `stocks.listing_date`

1. 進行中の会計年度（支払い未確定）は**常に除外**
2. 増配判定 `dps > prev*1.0001`、非減配判定 `dps >= prev*0.999`（浮動小数マージン）
3. **年度が連続しない箇所（歯抜け）でストリーク打ち切り**（地雷4）
4. ストリークが系列先頭または歯抜けに到達 → 一旦 `is_censored=1` 候補
5. `listing_date` があり `listing_year >= streak_start_year` → 真の開始と確定し `is_censored=0`
6. `override` があり `公表値 > 機械計算` → 公表値に置換（`is_censored=0`）
7. 残った `is_censored=1` は表示「N年以上 / N+」

**この計算モジュールはプロジェクト内に1つだけ**置く。DBアクセスと純粋計算を分離（旧 `back_findex/findex/metrics/dividend.py` が流用可能な完成形）。

### レート制限と検証コホートの位置づけ（重要）
全約4,000銘柄を一気に取得すると多くのサイトでレートリミットに当たる。
したがって**実装の検証は毎回全銘柄ではなく、このコホート（約30社）だけで行う**。
- fetch層は `--codes` でのサブセット取得を必須とする（requirements §9）
- 開発中は `--codes $(cohort)` で kabutan/yfinance を叩き、コホート分だけ取得して検算
- 全銘柄スキャンは設計が固まり golden が通ってから**1回だけ**実行する
- これによりレート制限を避けつつ、失敗モードを網羅した高速な反復ができる

## 7. 品質ゲート・不変条件

- **golden test**: 上表の確定値（花王36等）と機械計算＋overrideの結果が一致すること
- **不変条件テスト**: `streak_is_censored=1` の銘柄を**裸の数字で出力・投稿しない**（必ず「N+」）
- **境界異常テスト**: バックフィル接合後、前年比10倍超/1未満の行が残っていないこと
- これらが落ちている間は**X投稿を自動拒否**（数字が嘘＝最大リスク）

## 8. 実装手順（依存順）

1. 新スキーマ作成（`stocks` に `listing_date`/`founded_date`/`first_data_date`、`dividend_annual`、`streak_overrides`、`computed_metrics.streak_is_censored`）
2. 既存データ移行（`dividend_annual` の `source != 'events'` 行と `streak_overrides` は必ず移行。§requirements §10）
3. ストリーク計算モジュール（純粋関数＋打ち切り判定）＋ golden test
4. `listing_date` 取得（kabutanスクレイパー。容疑者band優先 → 全銘柄）
5. 判定式適用 → `streak_is_censored` 確定 → N+表示
6. 品質ゲート組み込み（投稿前チェック）

- 1989年境界13銘柄（東京瓦斯・大阪瓦斯・クレディセゾン等、非減配36年+）の確定値 → 各社IRファクトブック / 国会図書館デジタルコレクションで個別調査
