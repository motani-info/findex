## 0. 軌道修正の記録（なぜ本書が必要か）

旧PJ（`back_findex`）を参照した結果、「データ品質 = 2000年以前の配当履歴の欠落」と**問題が誤って狭められた**。
これは旧PJがその切り口で書かれていたためで、新 findex を立ち上げた本来の目的——**旧設計の前提のまま進む誤りを断つこと**——に反していた。

**是正**: findex の土台は「**あらゆる上場株式の、あらゆるデータが正確に保持されていること**」。
2000年以前の配当はこの大問題の **1セル**にすぎない。本書はこの正しい定款を起点に全体を設計し直す。

## 1. 定款（サービスのコアコンセプト）

> **あらゆる上場株式について、必要なデータを正確に保持し、足りない部分を常に把握できることを土台に、
> 独自指標（進化し続ける指標群）で多角的に分析し、Xユーザーの興味を引く切り口で投稿し続ける。**

3本の柱と、それを貫く1つの土台原則:

```
          ┌─────────────────────────────────────────┐
   柱3    │  発信   Xユーザーの興味を引く切り口で投稿     │
          ├─────────────────────────────────────────┤
   柱2    │  分析   独自指標による多角評価（指標群は進化）  │
          ├─────────────────────────────────────────┤
   柱1    │  基盤   全銘柄・必要データの正確な保持＋充足把握 │  ← 土台
          └─────────────────────────────────────────┘
土台原則:  正確でないデータは「無価値」ではなく「有害」。
          findex は自分が確証を持てない数字を、分析にも投稿にも一切流さない。
```

柱は積み上げ式。**土台（柱1）が崩れれば、その上の分析も発信も自動的に無価値（むしろ誤情報を拡散する有害物）になる。** ゆえに正確性は最優先制約であり、後段の都合で妥協しない。

**指標の数は定款の本質ではない。** 「18指標」は現時点のスナップショットにすぎず、指標は今後**追加・廃止・重み変更**される前提（`rule_versions` で版管理）。定款が約束するのは固定の指標集合ではなく、「**正確なデータの上で、進化する独自指標により多角評価する能力**」である。詳細は §5。

## 2. 「正確なデータ基盤」の定義

「正確」を曖昧にしないため、5つの測定可能な要件に分解する。全銘柄・全フィールドがこれを満たすことを目指す。

| 要件 | 意味 | 反例（あってはならない状態） |
|---|---|---|
| **網羅性** | 必要なフィールドに値が在る、または「無い」と明示されている | 値が黙って欠けている／NULLと未取得の区別がつかない |
| **正確性** | 値が一次情報（公的開示）と一致する | yfinanceのEPSキー揺れで誤った値が入る |
| **来歴** | すべての値が「どのソースか・いつ時点か・確信度」を持つ | 出所不明の値が混在する |
| **期間充足** | 各指標が必要とする年数の履歴が在る | 連続非減配20年指標を10年データで算出する |
| **自己認識** | システムが自分のデータ品質を銘柄×フィールド単位で把握している | 打ち切りや欠損に気づけず誤った数字を出す（旧実装の失敗） |

**自己認識が最重要**。旧実装は `listing_date` が0%で「データが切れている」ことに気づけなかった。
findex は「**知らないことを知っている**」状態を常に保つ。確証がなければ分析対象から外すか、不確実と明示する。

### 「完全」は目標であって前提ではない（実現可能性を直視する）
back_findex は「**全銘柄の全データを完全に揃えるのは不可能**」と結論している（無料で入手できないデータが存在する）。
この現実を踏まえ、定款の「正確に保持」は次の意味に定義する:

> **持てるデータは最大化する。持てないデータは「持てない」と正確に把握し、正直に扱う。**

つまり findex のゴールは「100%完全」ではなく（それは不能）、「**自分の充足状況を100%正確に把握していること**」。
- 揃うフィールド → verified にして全面活用
- 揃わないフィールド → insufficient/missing と明示し、依存指標を評価不能にする（黙って0や裸の数字にしない）
- 揃わない銘柄 → グレードDとして分析・投稿から外す（誤情報より「出さない」が正しい）

この線引きの前提として、「**どのフィールドが全銘柄分そろえられるのか**」を実証する必要がある（§7 の取得可能性スタディ D2.5）。

## 3. データ品質の問題空間（誤った狭め方の是正）

データ品質は配当だけの問題ではない。**リスク種別 × データ領域**のマトリクスで全体を捉える。
各セルが対処すべき課題であり、2000年以前の配当は左下の1セルにすぎない。

| リスク種別＼領域 | 銘柄マスター | 株価 | 配当 | 財務諸表 |
|---|---|---|---|---|
| **欠損**（値が無い） | 上場日が0% | 上場廃止銘柄の空白 | 歯抜け210銘柄 | **ROIC48%/FCF41%/信頼性37%** ← 最大の穴 |
| **不正確**（値が誤り） | 業種分類のずれ | 分割未調整 | 分割調整不一致(NTT等) | EPSキー揺れ・一過性利益の混入 |
| **期間不足**（履歴が短い） | — | 2000年floor | **2000年以前の配当** | 5年/10年CAGRに満たない若い銘柄 |
| **定義差**（基準が違う） | — | — | 公表値vs機械計算(±2年) | 会計年度の二重定義・連結/単体 |
| **ソース間不一致** | 名称表記 | ソース別終値差 | yfinance vs IR | EDINET vs yfinance |

**含意**:
- 規模で見れば**財務の欠損セル（universe の約半分）が最大**。配当の期間不足（約68社）より桁違いに広い。
- 旧実装・旧設計はこのマトリクスの一部しか見ていなかった。findex はマトリクス全体を体系的に塞ぐ。
- 個々の「地雷」（`back_findex/plans/02`）は、このマトリクスの個別セルの実例として有効。

### 真の根本原因（yfinanceは犯人ではない）
「yfinanceを軸にしたこと」は精度を下げた**一因**だが、**根本ではない**。根本原因は2つ:

1. **収集すべきデータの定義が不完全だった** — 「何が揃っていれば完全か」を仕様化していなかったため、欠損を測ることも気づくこともできなかった（旧 `stocks` テーブルが上場日を持たなかったのが象徴）
2. **欠損・誤りを検知できない設計だった** — 自己認識（§2）の欠如

→ **ソースを EDINET に替えるだけでは直らない。** 定義が不完全で検知できなければ同じ失敗をする。
設計は「**データ定義（データディクショナリ）= 完全とは何かの仕様**」から始め、ソース選定はその定義を満たす手段として後から決める（§4.0 → §4.1）。

## 4. データ完全性フレームワーク（中核設計）

全セルを統一的に塞ぐための仕組み。個別対処の寄せ集めにしない。**出発点はソースではなくデータ定義**。

### 4.0 データ定義（データディクショナリ）— すべての出発点
「完全とは何か」を先に仕様化する。これが無いと欠損を測れない（旧実装の根本欠陥）。
各フィールドについて以下を一意に定める **データディクショナリ** を、設計の最初の実体とする（D2）:

| 定義項目 | 内容 |
|---|---|
| フィールド名 / 意味 | 例: `equity_ratio`（自己資本比率） |
| 型・単位・値域 | REAL / %・0〜100 等。値域外は異常として弾く |
| 必須度 | 必須 / 任意（必須欠損はグレードを下げる） |
| 必要履歴年数 | 例: 連続非減配=可能な限り長く、10年CAGR=10年 |
| 主ソース / 照合ソース | §4.1 の階層から割当 |
| 検証ルール | 一次情報・golden との突合方法 |
| 依存指標 | このフィールドを使う指標（欠損時の影響範囲） |

カバレッジ（§4.4）・期間充足（§4.5）・品質ゲート（§4.6）はすべて、**この定義に対して**測られる。
「揃っているか」は常に「定義に対して揃っているか」を意味する。

### 4.1 ソース権威階層（定義を満たす手段。一次情報を主に、yfinanceを格下げ）
ソース選定は §4.0 の定義を埋めるための**手段**であって、根本対策そのものではない（§3 根本原因を参照）。
領域ごとに「主ソース（権威）」と「照合ソース」を定義する。**非公式で不安定な yfinance は主データから格下げし、フォールバック専用にする。**
領域ごとに「主ソース（権威）」と「照合ソース」を定義する。**非公式で不安定な yfinance は主データから格下げし、フォールバック専用にする。**

| 領域 | 主ソース（権威・一次情報） | 照合／補完 | フォールバック |
|---|---|---|---|
| 銘柄マスター | JPX 公式 | J-Quants 銘柄一覧 | — |
| 上場日・設立日 | kabutan | minkabu / EDINET沿革 | — |
| 株価 | J-Quants（公式） | — | yfinance |
| 財務諸表 | **EDINET XBRL（有報・一次情報）** | J-Quants statements | yfinance |
| 配当（現代） | J-Quants / EDINET | 各社IR | yfinance |
| 配当（2000年以前） | haitoukin / 各社IR | — | — |
| 検証用 正解値 | 各社IR・ダイヤモンドZAi等 | みんかぶ | — |

> 重要な転換: 旧実装は yfinance を主ソースにして欠損・誤りを量産した。
> findex は **EDINET（財務）・J-Quants（株価/配当）・JPX（マスター）の一次情報を主**とし、yfinance は埋め草に降格する。

### 4.2 来歴・as-of・確信度（全フィールドに付与）
すべての保持値に `source` / `as_of`（時点）/ `fetched_at`（取得時刻）/ `confidence`（検証済み・未検証・推定）を持たせる。
（データモデルへの反映は設計成果物 D3 で詳細化。現 `data-model.md` を改訂する。）

### 4.3 ソース間照合（reconciliation）
複数ソースに値があるとき、決定的に解決する:
1. 権威階層の上位を採用
2. 上位と下位の乖離が閾値超（例: 財務10%・配当 分割疑い10倍）→ `confidence=要レビュー` を立て、分析から除外
3. 解決履歴を記録（再現性）

### 4.4 カバレッジ＆品質マトリクスと銘柄グレード
銘柄 × フィールドごとに状態を持つ: `verified`（検証済み）/ `present`（在るが未検証）/ `insufficient`（期間不足）/ `missing`（欠損）。
これを集約し**銘柄データグレード**を付ける:

| グレード | 条件 | 扱い |
|---|---|---|
| A | 全必須フィールドが verified かつ期間充足 | 分析・投稿に全面採用 |
| B | present 主体・一部未検証 | 分析可・投稿は注記付き |
| C | 一部 insufficient/missing | 該当指標のみ除外して分析・投稿は慎重 |
| D | 必須が欠損・分析不適格 | ランキング除外・投稿対象外 |

カバレッジは可視化（ダッシュボード）し、品質改善の進捗を測る。

### 4.5 期間充足モデル（打ち切りの一般化）
各指標は「必要な最小履歴年数」を宣言する（例: 連続非減配=可能な限り長く・10年CAGR=10年・5年CAGR=5年）。
銘柄ごとに充足判定し、不足なら「**N年以上（N+）／不足**」と明示する。
**これは配当ストリーク専用ではなく、全指標に適用する一般則。** 2000年以前の配当打ち切り（pre2000-data.md）はこのモデルの一事例。

### 4.6 データ層の品質ゲート（silent never）
分析・投稿の**前段、データ層で**ゲートする。
- グレードDは分析対象外
- `confidence=要レビュー`・`insufficient` のフィールドは該当指標を「不足」とし、**黙って0点や裸の数字にしない**（rules.yaml の現Nullポリシー「None→0点」は本原則と矛盾するため D4 で再設計）
- 投稿層のgolden gateは最後の砦であって、唯一の砦にしない

### 4.7 継続的検証（多フィールドgolden）
検証を配当ストリークだけでなく**多数のフィールド**へ拡張する。
- 各社IR・公的開示から既知の正解値（売上・配当・自己資本比率等）を golden セット化
- 定期的にDB値と突合し、乖離を検出してレポート
- 検証スイートが赤い間は投稿停止（土台原則の自動執行）

### 4.8 取得できないデータの保管と手動バックフィル（要調査・検証）
バルク（API/スクレイピング）で取得できないデータをどう埋め、どう保管するかを定義する。
**鍵は「自己認識が手動収集を現実的にする」ループ**：

```
カバレッジ追跡(§4.4) ──→ 欠損・不足の (銘柄×フィールド) を列挙 = バックフィル・ワークリスト
       │  ※マスターデータ(上場日・設立日)が gap set を特定可能にする
       ▼
優先度付け（ランキング/投稿に効く銘柄・フィールドから）
       ▼
1銘柄ずつ収集（各社IR・HP・ファクトブック・国会図書館 等）
       ▼
来歴付きで保管（バルク値と区別・上書き禁止）
       ▼
検証 → カバレッジ更新（穴が1つ埋まる）
```

**例（ユーザー提案）**: 2000年以前上場の会社が100銘柄あるなら、`stocks` の上場日・設立日で**その100銘柄を特定**し、1社ずつIR/HPから不足データを収集する。4000件一括は無理でも100件なら現実的。

**保管モデルの原則**（詳細仕様と方式選定は D2 / D2.5）:
- すべての値に来歴を持たせる（§4.2）: `source` / `confidence` / `source_url` / `as_of` / `collected_at`
- **手動・IR値はバルク値より優先**し、**バルク再取得で上書きしない**。優先順位 `manual ≥ ir > 一次バルク(EDINET/J-Quants) > フォールバック(yfinance)`
- 手入力は誤りやすい → `source_url` 必須、`confidence` で `manual_verified` / `manual_unverified` を区別、可能なら golden 突合
- ワークリストの進捗を追跡（残り何件か＝充足の自己認識そのもの）

> back_findex の `dividend_annual.source`（events/haitoukin/ir/manual）と `streak_overrides` は、この保管モデルの**配当ドメインでの先行実装**。これを全フィールドに一般化する。

**要調査・検証（D2.5に集約）**:
- 各フィールドの gap set の規模（何銘柄が手動対象になるか）
- 1銘柄ずつの収集が現実的な工数か（フィールド別の入手性）
- 保管モデルの方式選定: ドメイン別 `source` カラム方式 vs 汎用 provenance/override 層
- 手動データの正確性をどう担保するか（出典必須・二重チェック・golden）

## 5. 分析層：進化する独自指標システム

指標は**固定ではなく進化する資産**。現時点は18（流用 `rules.yaml` は統廃合済みの16=v3）だが、数は本質でない。
設計が担保するのは「指標を**安全に増減・改訂できる仕組み**」である。

### 指標システムの要件
- **版管理**: 指標の追加・廃止・重み変更は `rule_versions` に版として記録。過去スコアは当時の版で再現可能
- **データ契約**: 各指標 = { 依存フィールド, 最低品質（grade/confidence）, 最小履歴年数, 不足時の挙動 }。指標を足すときはこの契約を必ず定義する
- **データ定義との連動**: 新指標が要求するフィールドは §4.0 のデータディクショナリに追加され、カバレッジ・取得可能性の対象になる（「指標は増やせるが、裏付けデータが揃うかは別問題」を可視化する）
- **不足時の挙動は「黙って0点」を禁止**: 「不足（評価不能）」として扱い、スコアと並べてデータ品質を提示する。グレードC銘柄は欠損指標を除いた部分採点とその旨を明示

> 数（16/18）の確定は D4 で行うが、それは「現時点のスナップショット」を決めるだけ。
> 定款・アーキテクチャは指標数に依存しない。

## 6. 発信層：Xユーザーの興味を引く切り口

定款の第3の柱。「正確さ」と「面白さ」を両立させる設計を、出力層の付け足しではなく**一級の設計対象**として扱う。

**大原則**: 投稿する数字は必ずデータ層の品質ゲートを通った検証済み事実のみ。`as_of` を添える。面白さのために正確性を犠牲にしない。

**切り口（フック）の体系**（詳細は設計成果物 D5）:
- ランキング（連続増配TOP・隠れ増配・高利回り×高安全）
- 意外性（「実は36年増配の地味株」「高利回りなのに減配リスク低」）
- 逆張り（割安に放置された優良増配株）
- 危機耐性（リーマン・コロナを越えた銘柄）
- 季節性（権利確定月カレンダー）
- 比較（セクター内対決・大型vs中小）
- 異常検知（増配率の急加速・配当性向の急改善）

**エンゲージメント学習**: 投稿のテーマ別反応（いいね/RT/インプレッション）を記録し、伸びる切り口を優先するフィードバックループを設計する。
